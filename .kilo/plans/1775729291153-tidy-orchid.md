# Plan: Show backend 409 delete dependency message for category/priority

## Goal

When deleting a `TodoCategory` or `TodoPriority` that is still used by active tasks, surface backend `409 Conflict` payload

```json
{ "message": "Entity cannot be deleted because it has 1 dependent TodoTask record(s)" }
```

as a user-visible error notification in the dashboard.

## Current behavior and root cause

- Delete flows already catch errors and show feedback in:
  - `src/modules/tasks/components/DashboardCategoriesPanel.vue`
  - `src/modules/tasks/components/DashboardPrioritiesPanel.vue`
- Error text is generated through `toAppErrorMessage` in `src/shared/utils/error-message.ts`.
- `toStatusAwareMessage` already preserves backend message for `409`, **but** `toUserMessage` currently reads only:
  - `messages: string[]`
  - ProblemDetails `detail` / `title`
- Backend returns `message: string` (singular), so frontend falls back to generic conflict text.

## Implementation steps

1. Update error payload parsing in `src/shared/utils/error-message.ts`:
   - Extend `toUserMessage` to support singular `message: string` payloads.
   - Keep existing support for `messages[]`, `detail`, and `title` for backward compatibility.
   - (Optional robustness) If body is a non-empty string, return it directly.
2. Keep `toStatusAwareMessage` logic for `409` as-is, so once `message` is parsed it is shown automatically.
3. Add tests for regression safety:
   - New test file: `tests/shared/error-message.test.ts`.
   - Verify `toUserMessage({ message: '...' })` returns that message.
   - Verify `toAppErrorMessage(new ApiClientError({ status: 409, body: { message: '...' } }))` returns backend message.
   - Verify existing `messages[]` behavior still works.
4. Validate via test run:
   - `npm run test -- tests/shared/error-message.test.ts`
   - Optionally run full suite: `npm run test`

## Expected outcome

- Attempting to delete in-use category/priority shows exact backend conflict message in feedback toast.
- No component-level delete flow changes are required beyond improved centralized error parsing.
- Existing error handling paths remain compatible.
